## A. Fixed (verified: typecheck clean + npm test 30/30, new regression test added)

### A1 — `[SHALLOW]` safety gate silently disabled  ·  HIGH  ·  real data  ·  `evals/trust-checks.ts`
The materialised transcript (`toLooseTranscript` → `LooseTurn`) **dropped the per-turn `note`
field**. `reviewer.computeReadQuality` reads `t.note` to spot `[SHALLOW]`-flagged answers; with
`note` always `undefined`, a thin/garbled session that still shipped a confident high-magnitude axis
read **no longer tripped `OVERDIAGNOSIS_ON_THIN`** (and `read_quality` flipped partial→full).
`[SHALLOW]` notes occur in production transcripts (e.g. `logs/june/2026_Jun02_17-35-b4b209e3`).
- **Fix:** added `note?: string` to `LooseTurn`; copy it through in `toLooseTranscript`.
- **Test:** new red→green case in `scripts/test-trust-checks.js` — substantive answers (clear the
  token floor) each carrying a `[SHALLOW]` note must still fire `OVERDIAGNOSIS_ON_THIN`. The existing
  case 5 used genuinely short answers, so it passed even with the bug — this new case isolates `note`.

### A2 — QUESTION_INTEGRITY gate weakened  ·  LOW (latent)  ·  `evals/trust-checks.ts:312`
Conversion added an `&& q.alias` short-circuit, suppressing the hard fail the original raised for a
`source:"generated"` served question with **no alias** (in a session whose bank has aliases). The
guard was only there to satisfy `Set<string>.has`.
- **Fix:** `!bankAliases.has(q.alias ?? "")` — restores the original failure (empty alias ∉ bank →
  fail) without a banned cast.

### A3 — free planner fallback became a paid live call  ·  MEDIUM  ·  cost rule  ·  `backend/api/handlers/plan.ts:123`
Original `session.focusPointsResult.focus_points` **threw** when `focusPointsResult` was `null`
(caught → free `"(planner failed)"` fallback, no model call). Conversion's `?.` swallowed the null,
so `planTurn` ran a **live (paid) plan** with `focusPoints:undefined`. Reachable when a session
reaches `POST /api/plan` with a pending answer but no focus result (focus-gen failed/aborted, or a
rehydrated/replayed session).
- **Fix:** explicit `if (!session.focusPointsResult) throw …` inside the same `try`, then direct
  access. Faithful (null → throw → existing catch → free fallback) and type-narrows; no `?.`, no cast.

## C. Found by the LIVE run, NOT by the review — a whole class the review missed  ·  HIGH  ·  FIXED

The agent review checked **import specifiers** and proved them clean. It did **not** catch
**string-literal `.js` file paths** (read via `fs.readFileSync` / matched via `fs.existsSync`), because
those aren't imports. Three such stale paths survived the conversion — and the first one **broke the
live pipeline entirely** (every gate/smoke run died at "pipeline incomplete"). I only found them by
trying to run gate #3. Lesson: the robot review was necessary but **not sufficient** — the paid
behaviour proof earned its keep.

- **C1 — `smoke-test.js` pre-flight (`PROMPT_SRC_MAP`, 6 paths):** `readFileSync("backend/engine/generate.js"…)`
  → ENOENT → 6 unit checks fail → smoke aborts before E2E (exit 1). **Broke gate + smoke completely.**
  Fixed `.js`→`.ts`.
- **C2 — `smoke-test.js` CLI spawn:** spawned `node cli.ts` (root) but the CLI is `backend/cli.ts`
  (`Cannot find module …/cli.ts`). The conversion changed `["cli.js"]`→`["cli.ts"]` but dropped the
  `backend/` path. Fixed to `path.join(__dirname, "backend", "cli.ts")`.
- **C3 — `backend/engine/pipeline-lock.ts` (`PATH_META`, ~28 paths):** keyed by `backend/engine/*.js`
  etc.; `existsSync` **silently skipped** all of them, so the pipeline fingerprint no longer covered the
  core engine/api source (fails safe, no crash — but a silent coverage gap). Fixed `.js`→`.ts`.
- *Not fixed (throwaway phase-verification scaffolding, not in the gate/test path):*
  `scripts/batch-l-verify.js`, `scripts/manual-qa-verify.js` carry one stale `.js` path each — flagged,
  left alone (surgical).
