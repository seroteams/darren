## Built — 2026-07-10

- **Store** — `feedback_notes` gained nullable `run_id` + `verdict` columns + an index (migration `0013_abandoned_zodiak.sql`, generated by drizzle-kit; the boot self-migrates — applied to local Neon during verification). One feedback store, one inbox, as decided.
- **Service** — `submitVerdict` (test-first, 6 tests red → green, 16/16 in the file): validates yes/no + run id, comment optional and capped; **upsert per run** — a re-tap or a late comment updates that run's row, never a duplicate (an empty re-tap can't wipe an earlier comment). Anonymous allowed: a guest's tap counts.
- **Route** — `POST /api/v1/feedback/verdict`, origin-guarded, deliberately no login wall (mirrors the client error-report route). The plain-note route stays login-required and untouched.
- **Briefing card** (`briefing.js`, the live customer briefing — shows for guests too, hidden on the scripted test lane): one question, Yes/No ghost buttons, saves on tap ("Thanks!"), optional one-line comment appears after ("Noted — thanks!"). Failures soft; ignoring it costs nothing.
- **Inbox** (`admin-feedback.ts`): verdict rows show a colour pill ("would change: yes/no"), the run reference (short id, full on hover) in the Screen column, and the comment; plain notes render exactly as before.
- **Verified live as a real guest** (customer app on 3085, anonymous, cloned ownerless run): card rendered on the briefing → tap Yes saved → comment attached → re-tap No updated. **DESTINATION check:** direct SQL after three POSTs found **exactly one row** — `{run_id: vk3-verify-tap, verdict: no, message: "walk verification comment", user_id: null}`. Inbox showed it with pill + run ref. Test artifacts then deleted (DB back to production-true).

## Test scenarios — for the product owner
1. **The tap** — finish a prep (guest or test account) and reach the briefing. You should see one plain question with Yes/No. Tap one, add a short comment, done — under 10 seconds. ❌ Not OK if it feels like a survey or interrupts reading.
2. **It landed** — open the superadmin feedback inbox. Your tap should be there, tied to that run. ❌ Not OK if it shows with no way to tell which 1:1 it came from.
3. **Ignorable** — run another prep and ignore the question completely. Nothing should nag, block or re-ask. ❌ Not OK if it demands an answer.
