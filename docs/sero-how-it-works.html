02 · The big picture

Two things in one repo

It helps to see Sero as a product wrapped in a workshop. Same engine powers both.

The product — a guided 1:1

A manager types a few rough notes ("she's been quiet"). Sero suggests focus areas, writes a prep briefing, builds a question bank, runs the live conversation (scoring as it goes), and writes a final briefing with truths and next actions. This is what a manager touches.

The workshop — keeping it honest

Around the product sits a thick layer of build & QA tooling: a run library (every past run), compare two runs, verdict each one, a regression safety check, gates, and an offline eval loop. This is how the team trusts the engine before it ships.

One engine, two layouts, every run logged. The manager run can be the classic screen-by-screen flow or the new one-page grow-down — identical answers and briefing either way. Everything is filesystem-backed; there's no database.
03 · The pipeline

Six stages, each with its own model

0

Intake

Name, role, seniority, meeting type, rough notes. No AI.

1

Focus areas

2–5 things to explore, from a catalogue of 21. nano model.

1b

Preparation

The game plan: core issue, opener, listen-for, avoid. Full model.

2

Question bank

8–12 tailored questions. mini model.

3

Live questioning

Score + re-plan after every answer. Full model, per turn.

4

Final briefing

Headline, truths, next actions, watch-fors. mini model.

  • Right model for the job: a fast gpt-5.4-nano for focus areas, mini for the bank and evaluation, the full gpt-5.4 for the prep briefing and the live planner. All set in config/models.json.
  • Optional stage 5 — phrase library: on eligible meetings, Sero harvests the wording the employee actually used, to keep future questions in their language.
  • Every stage streams to the screen so you can watch it think, and logs its inputs, prompt, and response to disk.

05 · Meeting types & arcs

Five kinds of 1:1, each with a shape

Every meeting type has an arc — a planned run of stages, like acts in a play, each with a target number of questions. The planner moves the conversation through it.

Bi-weekly check-inCasual catch-up, 15–20 minpulse → friction → momentum → liftRelational: no performance-judging topics
Performance & feedbackDirect, adult-to-adultself-read → evidence → gap → cause → commit—
Growth & career planForward-looking, 30–45 minanchor → aspiration → gap → investment → commitmentIts own final-briefing prompt
Something feels offGentle, observation-firstlanding → observation → underneath → supportRelational: no performance-judging topics

Arcs are browsable & editable from the "Meeting arcs" page (each shows tone, anti-patterns, and per-stage question targets). And the relational rule — no competency topics in a casual check-in — is now enforced at four layers: input filter, prompt, the focus gate (FOCUS_ARC_LEAK), and the question-layer gate (QUESTION_ARC_LEAK).
06 · Focus areas

What the meeting is really about — 21 on the menu

Sero picks from a fixed catalogue of 21 focus areas (in focus-points.json), each with worked label examples so the wording fits the level:

Wellbeing — 3

Workload & capacity, energy & wellbeing, team connection. How the person is actually doing.

Topic — 7

Priorities, blockers, growth, feedback, recognition, role clarity, manager support. The work worth airing.

Competency — 11

Quality, speed, ownership, communication, reliability, judgment, impact, decision speed, technical problem-solving, stakeholder engagement, delegation. The evaluative ones — off-limits in relational meetings.

  • Each suggestion is a signal (drawn from your notes) or a best practice (sensible for this role anyway) — and carries a reason.
  • If your notes have real content, at least one suggestion must come from them — Sero can't ignore what you wrote.
  • You start with nothing selected and opt areas in; your picks (/api/selected-focus) steer the question bank.
07 · The preparation briefing

The manager's game plan

Once focus areas are chosen, Sero writes a short strategy — always the same six parts:

Core issue

The one thing this meeting is really about, in a sentence.

Opening question

How to start so it opens up instead of stalling.

Listen for

Three behavioural cues — pauses, deflections, energy shifts.

Avoid

Two pitfalls likely to shut this conversation down.

Good outcome

What "this went well" looks like, concretely.

Suggested action

One thing the manager could offer or commit to.

Quality control built in: the briefing is validated before you see it; if it fails (wrong shape, missing parts) Sero retries once. It refers to people by name, not job title, in plain language — a standing rule.
08 · The question library

3,807 questions, each its own small file

Every question Sero has generated and kept lives as a tiny YAML file in questions/. Each records what it says and how it behaves:

What it says

The question text, a short label, a description of what it surfaces, and its purpose: wellbeing, topic, or competency.

How it behaves

Its arc stage, which axes it can move and by how much, and where it came from (original, generated, reworded…).

  • The index (questions/_index.json) is the master list — rebuilt with npm run rebuild-question-index when files change.
  • Openers & seeds are kept separately: research-backed openers plus per-meeting-type intro questions used to start each run.
  • The dedup gate stops near-identical questions twice in a session — it compares the actual words, not the filename.
  • Questions stay in their own run: ones invented mid-meeting save to a separate runtime area, so they can never leak into someone else's run.
09 · Live questioning

The meeting — planned one turn at a time

A run has a budget of 9 questions: 4 intro + 5 dynamic. After each answer, the planner recomputes a handful of runtime signals and decides what's next.

What it tracks each turn

How far through the arc it is, which stages remain, how many drills in a row at one spot, off-arc tangents used, whether this is the final turn, and the last answer's scores.

What it decides

Stay on this thread, drill deeper, or advance the arc — adding, rewording, or dropping upcoming questions to fit.

Hard caps in code

≤2 drills in a row at one spot, ≤2 wellbeing clarifiers in a row, ≤1 off-arc tangent. When questions left ≤ acts left, it must advance.

  • Shallow answers don't count as good news. A one- or two-word answer (or a skip) gets any positive scores zeroed — "fine" is not evidence things are fine.
  • Grounding gate: a brand-new planner question must quote a real premise from the notes or transcript — invented premises ("you're up for promotion") get dropped.
  • Broken questions get a safe fallback. If a "you said…" follow-up can't lift a clean phrase, it's swapped for a sound stem instead of shipping garbled.
10 · Axes & scoring

Four needles, all starting at zero

Wellbeing · Engagement · Clarity · Growth. Every run starts all four at 0 — neutral. Each answer nudges them by a fixed step (−3, −1, 0, +1, or +3); scores accumulate and clamp to ±10 (the bars visually max out at ±6 with an "offscale" badge beyond).

−10 … −5 act / running hot
−4 … −2 watch
±1 weak
+2 … +4 watch
+5 … +10 strong / thriving
−100 · start here+10
±0–1 — weak signal, not actionable. Ignore.
±2–4 — worth watching over the coming weeks.
±5–7 — a real pattern. Act on it.
±8–10 — a defining signal of this conversation.
Why neutral start matters: nothing is assumed about how someone's doing — a positive or negative read has to be earned by what they actually say. The direction (green up / red down) and the magnitude both come only from evidence in the answers.
11 · The final briefing

What you walk away with

After the last question, Sero reads the whole transcript plus the axis scores and writes the wrap-up:

Headline & summary

One headline (≤22 words), 2–3 bullets, and a short paragraph on how well the person was actually understood.

Axis read-outs

For each axis: what the score means in this conversation and what tone to take about it.

Two brutal truths

One safe to share with the employee. One manager's-eyes-only — never mixed into anything employee-facing.

Three next actions

Each with a "when" attached, so they don't evaporate.

Watch-fors

Two observable signals to keep an eye on before the next 1:1.

Honesty switches

If the meeting was thin, the briefing leads with that. If a high score rests on one or two mentions, its confidence is downgraded.

"Thin" has a precise meaning: fewer than 3 substantive turns, or under 60% of turns carrying real signal. Then the headline names the read-quality first, the summary shrinks, and one of the next actions is "have the conversation again". Disengagement is the one call Sero won't make on weak evidence — it stays inconclusive rather than risk a wrong label.
12 · Role profiles & vocabulary

Sero knows the job, not just the person

The first time a role + seniority appears, Sero generates a role profile once and caches it in data/role-profiles/. Later runs reuse it for free. 15 are on file (Account Exec, Backend Eng, CSM, Head of Product, UX Lead, SRE, Ops Coordinator…).

What's in one

A summary, the role's known challenges, recommended question themes, the terminology people in that job use, things to listen for, and things to avoid.

What's NOT in one

Anything personal. Profiles are built from title + seniority only — safely shared across everyone holding that title. The cache key is the exact title (a Staff SRE never gets a Backend Eng profile).

  • Profiles feed the question bank (questions sound native to the job) and the prep briefing (advice fits the role).
  • The terminology is surfaced to you — as the "language of this role" glossary in a one-page run, and on the Job Lexicons page.
  • In relational meetings the evaluative parts are filtered out, with a gate (ROLE_PROFILE_ARC_LEAK) to catch leaks.
13 · Job lexicons NEW

Browse — and add to — each job's words

A new left-rail page lists every role's vocabulary: the AI's terms from each role profile with plain meanings. You can also teach Sero your own job words.

Browse anytime

No active meeting needed. Open "Job lexicons" and see each title with its words — what "snagging" or "handoff" means in that world.

Add your own

Add a term to any role ("Standup → quick daily sync"). Yours are marked "yours" and saved to a separate overlay file (e.g. …--mid-level.overlay.json) — never overwriting the AI's, removable in one click.

  • Your words show up in the "language of this role" glossary during a one-page run.
  • Distinct from the older phrase library (which harvests wording from one past session). Job lexicons are permanent, per-role, and editable.
  • Coming next: your words will feed the live assistant so its questions speak the team's language.
The point: Sero should sound like it works on your team, not like a generic HR tool — and you steer that, one word at a time.
14 · The safety gates

Eleven hard rules that fail the build

npm run gate replays human-approved "golden" cases through the live pipeline. Any one of these failing means the change doesn't ship — defined in evals/trust-checks.js.

GateWhat it stops
PRIVATE_NOTE_LEAKThe manager's private judgments surfacing word-for-word where the employee could see them.
OVERDIAGNOSIS_ON_THINConfident, high scores from a thin, evasive meeting.
FOCUS_ARC_LEAK / QUESTION_ARC_LEAKA relational meeting sneaking in performance-judging topics — caught at the focus layer and the question layer.
ROLE_PROFILE_ARC_LEAKThe same sneaking, but via the role profile's evaluative content.
CROSS_SESSION_QUESTION_LEAKA run serving a question or vocabulary from someone else's session.
ENGINE_VOCAB_LEAK / ROLE_PROFILE_VOCAB_LEAKInternal machinery words (placeholders, field names) leaking into what users read.
WRONG_MEETING_TYPEA question bank that ignores the meeting's arc.
SCHEMA_INVALIDStructurally broken output — missing parts, scores out of range.
QUESTION_INTEGRITYDuplicate questions, forbidden phrasings, debug text, or questions from nowhere.
15 · Engine trust gates NEW

Six new protections, one per problem found

A review of past runs turned up five recurring engine bugs. Each was fixed and locked behind a check so it can't come back:

#ProtectionWhat it now guarantees
1Question isolation
CROSS_SESSION_QUESTION_LEAK		Mid-meeting questions save to a separate runtime area — they can never reappear in an unrelated run.
2Honest follow-up stemsA "you said…" follow-up must quote a real, contiguous phrase from the answer; no clean phrase → it's skipped, not faked.
3Grounded premises
UNGROUNDED_PREMISE		Every new planner question cites a short verbatim quote from notes/transcript, or is marked an open question. Invented premises get dropped.
4Relational-arc at the question layer
QUESTION_ARC_LEAK		Bi-weekly and "feels off" can't generate, pick, or add a competency question — protection sits where questions are chosen.
5Axis honesty
AXIS_SILENT_SESSION		Carried-forward questions keep their scoring power, so a real conversation no longer ships every axis as "didn't come up".
6Confidence honesty
RULE_ECHO_MEANING		A high score on ≤2 mentions gets its confidence downgraded; meanings that just echo a prompt example are flagged, never silently rewritten.
The through-line: say what's actually true. Don't borrow questions, invent premises, fake confidence, or paper over a problem.
16 · Guardrails & advisory checks

Three nets, in order

Prompts ask the model to behave; code enforces what really matters; the gates catch anything that slips. Some checks are advisory — they warn and log rather than fail the build.

Prompt-level rules

Focus-area labels can't start with question words. A banned-phrase list keeps out consultant-speak ("pulse check", "hygiene"). Notes with real content must yield ≥1 note-driven focus area. ≥half the bank must reference this specific person.

Code-level enforcement

Drill/clarifier/tangent caps. Shallow-answer zeroing. The grounding gate. The arc-leak filter. Safe fallbacks for broken questions. These don't ask — they enforce.

Advisory (warn + log)

UNGROUNDED_PREMISE, AXIS_SILENT_SESSION, RULE_ECHO_MEANING — surfaced for us to inspect, not hard-blocking yet. 14 named checks in total: 11 hard-fail + 3 advisory.

The standing rule

No silent masking. When the model produces something off, Sero flags it — it never quietly rewrites text to hide a problem. Surfacing the truth beats a clean-looking lie.

17 · Logs

Every run, fully on file

Each run gets a folder: logs/<month>/<run-id>/. Every stage keeps three things: the inputs it got, the exact prompt sent, and the response back.

Per stage

inputs.json, prompt.md, response.json for each stage, plus one file per turn during questioning.

Per run

transcript.json (all Q&A), axis-state.json (final scores), notes.md, session state, and the run's cost in tokens and dollars.

The fingerprint

pipeline-lock.json records exactly which prompts, code, and models produced this run — so any two runs can be diffed.

Two things to remember: response.json is the raw model output, before the guards run — judge the product on what shipped, not the raw draft. And reviewed runs carry a review.json verdict (the QA cockpit writes it; it never touches the run itself).
18 · The QA cockpit under-documented

How the team keeps the engine honest

A big slice of the app is internal tooling for reviewing runs and turning what we learn into prompt fixes. The loop:

Library — the inbox

Every finished run, newest first. Filter by verdict (Unreviewed / Keep / Fix / Block), search, and archive. A progress bar tracks how many are reviewed.

Review — the verdict

Read a run end-to-end and mark 8 dimensions Pass/Fail — audience resonance, clarity, appropriate challenge, actionability, accuracy, tone, completeness, factual precision — then Keep / Fix / Block. Saved to review.json.

Compare — side by side

Load two runs together to see what a prompt change actually moved: headlines, questions, notes, and the four axis reads with delta bars.

Suggest-fix

With a verdict recorded, ask the model for a minimal edit to one stage's prompt. Display-only — a human applies it by hand.

Regression — instant safety

Re-checks saved "key runs" against the current engine with no AI calls — same logic as npm run replay. Catches a prompt change breaking a safety rule in seconds.

Notes & debrief

A right-rail notes panel tags observations to the stage/turn they happened on. The run-debrief shows time, cost, and turn stats, and hands you a ready QA prompt.

Why it's here: Sero earns trust by being audited constantly. Most of these screens (Library, Compare, Regression, Review, suggest-fix) are for the team; the manager only ever sees the run itself.
19 · Testing & the eval loop

What proves it still works

npm run gate

The trust gate. Replays 8 golden scenarios through the live pipeline and runs all hard checks. Pass/fail. Run before calling any change done. ~$3 full, ~$0.35 a single case.

npm run smoke

End-to-end check. Pushes a whole scenario start to finish to prove the plumbing connects.

npm run eval

The quality lens. Replays batches and scores soft qualities — question specificity, thread-follow, whether scores match the transcript.

  • The prompt-notes loop: human-written rules in *.notes.yaml (must/must-not, length & count ranges) are checked against output offline. When one fails, we read why, tweak the prompt, and re-check — without burning API.
  • Free, offline nets: npm test, the regression replay (5 fixtures: jordan-kim, leak-devon, marcus-lee, maya-chen, thin-sam), and offline trust-check tests. These are the default.
  • Cost discipline: no paid API run without an explicit yes and a stated cost first. No self-certifying — a prompt/engine change isn't done until the gate has actually run and been reported.
20 · The app & the CLI

Two front doors, one engine

The web app

A left-rail sidebar holds every page — Home · New session · Library · Compare runs · Regression · Phrase library · Job lexicons · Meeting arcs (+ Guide in dev) — collapsing to icons when idle. A run shows a stage breadcrumb and "Step N of 7"; a "?" opens a glossary; a thinking orb plays during AI stages.

The CLI

npm run cli runs the same pipeline in a terminal — mostly for testing and replays. It shows a changelog banner when prompts/code/models changed since the last run, and asks for a 1–5 rating at the end.

  • In development: npm run dev starts the API on port 3001 and the page on 3000, together. (Under preview tools, run them separately — the port injection clashes.)
  • Sessions survive restarts — a half-finished meeting can be saved and resumed.
  • Light protections: requests only from your own machine; new-session rate limits stop runaway loops.
21 · How we build

The Darren Method

Anything bigger than a one-file tweak is planned and built one careful step at a time:

1

Plan

A folder under docs/todo/<name>/: a PLAN.md plus one file per phase, each ending in test scenarios.

2

Baseline

Run the gate before touching anything, so old failures aren't blamed on new work.

3

One phase

Build exactly one phase. No racing ahead.

4

You test

The product owner walks the scenarios and green-lights. The builder never self-certifies.

5

Commit & next

Approved work commits immediately. Only then does the next phase start.

One board tracks it all: SERO_BOARD.md at the repo root is the single source of truth — Now / Next / Parked / Cut / Done, with commit hashes. Finished folders move to docs/todo/done/; mid-phase ideas get parked, never bolted on. Right now several builds (engine-trust-gates, role-profiles, one-page run, job lexicons) sit in Now — committed, awaiting your walkthrough.
