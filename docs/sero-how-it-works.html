01 · The big picture

From scribbled notes to a great 1:1

A manager types a few rough observations — "she's been quiet lately", "we need to talk about growth". Sero does the rest:

Before the meeting

Sero suggests focus points (what to actually explore), writes a preparation briefing (how to open, what to listen for, what to avoid), and builds a bank of tailored questions.

During the meeting

The manager works through questions one at a time. After each answer, Sero re-plans on the fly — picking the next best question and quietly scoring what the answer revealed.

After the meeting

Sero writes a final briefing: a headline, what was learned on each axis, two "brutal truths" (one shareable, one manager-only), three next actions, and things to watch for.

One engine, two front doors. The same pipeline runs behind a web app (what you normally use) and a command-line version (used mostly for testing). Every run is logged to disk in full.
02 · The pipeline

Six stages, in order

0

Intake

Name, role, seniority, meeting type, and the manager's rough notes. No AI yet.

1

Focus points

AI picks 2–5 things worth exploring, from a fixed catalogue of 21. Manager ticks the ones they want.

1b

Preparation

AI writes the game plan: core issue, opening question, what to listen for, what to avoid.

2

Question bank

AI builds 8–12 questions tailored to this person, this meeting type, this moment.

3

Live questioning

The meeting. After every answer, AI scores it and re-plans the remaining questions.

4

Final briefing

AI sums up the whole conversation: headline, truths, next actions, watch-fors.

Plus an optional stage 5: on some meeting types, Sero reviews the session for wording the employee actually uses ("lexicon review") so future questions sound more like their world, not corporate-speak.

Each AI stage uses the model best suited to the job — a small fast one for focus points, bigger ones for the briefing and the live planner. Stages stream to the screen so you can watch them think.

03 · Meeting types & arcs

Five kinds of 1:1, each with a shape

Every meeting type has an arc — a planned sequence of stages the conversation should move through, like acts in a play. The question planner respects the arc.

TypeFeelArc (the acts)Special rule
Bi-weekly check-inCasual catch-up, 15–20 minpulse → friction → momentum → liftRelational: no performance-judging topics
Performance & feedbackDirect, adult-to-adultself-read → evidence → gap → cause → commit—
Growth & career planForward-looking, 30–45 minanchor → aspiration → gap → investment → commitmentHas its own final-briefing prompt
Something feels offGentle, observation-firstlanding → observation → underneath → supportRelational: no performance-judging topics
Onboarding check-inFor new joiners, 15–25 min(arc still being finished)Newest type
Why the "relational" rule matters: if a casual check-in suddenly asks about output quality or speed, it feels like an ambush performance review. So the two relational types are banned from using competency topics — enforced three times over (in the input, in the prompt, and by a hard gate that fails the build if it ever leaks).
04 · Focus points

Deciding what the meeting is really about

Sero doesn't invent topics from thin air. It chooses from a fixed catalogue of 21 focus points, in three flavours:

Wellbeing (3)

Workload, energy, team connection. How the person is actually doing.

Topic (7)

Priorities, blockers, growth, feedback, recognition, role clarity, manager support. The work stuff worth airing.

Competency (11)

Quality, speed, ownership, communication, judgment… The evaluative ones — off-limits in relational meetings.

  • Each suggestion is marked as a signal (drawn from what you wrote in your notes) or a best practice (sensible for this role and meeting anyway) — and comes with a reason.
  • If your notes contain real content, at least one suggestion must come from them. Sero can't ignore what you wrote and serve only generic advice.
  • You start with nothing selected — you opt focus points in. Nothing gets explored without your say-so.
05 · The preparation briefing

The manager's game plan

Once focus points are chosen, Sero writes a short strategy for the conversation. It always has the same six parts:

Core issue

The one thing this meeting is really about, in a sentence.

Opening question

How to start so the conversation opens up instead of stalling.

Listen for

Three behavioural cues to watch for — pauses, deflections, energy shifts.

Avoid

Two pitfalls likely to shut this particular conversation down.

Good outcome

What "this went well" looks like, concretely.

Suggested action

One thing the manager could offer or commit to.

Quality control built in: the briefing is checked against a validator before you see it. If it fails (wrong shape, missing parts), Sero retries once automatically. And it refers to people by name, not job title, in plain language — that's a standing rule.
06 · The question library

~3,750 questions, each one a small file

Every question Sero has ever generated and kept lives as its own little YAML file in the questions/ folder. Each one records:

What it says

The question text, a short label, a description of what it's trying to surface, and its purpose: wellbeing, topic, or competency.

How it behaves

Which arc stage it belongs to, which axes it can move and by how much, and where it came from (original, generated mid-meeting, reworded…).

  • The index (questions/_index.json) is the master list — rebuilt with npm run rebuild-question-index when files change.
  • Openers and seeds: a small set of research-backed opening questions, plus per-meeting-type intro questions, are kept separately and used to start each run.
  • The dedup gate stops near-identical questions being asked twice in a session — it compares the actual words, not just the file name (a real bug it caught: the same follow-up asked on back-to-back turns).
  • The eligibility gate blocks questions that match a meeting type's forbidden patterns, and logs every rejection (for us to inspect — never shown to users).
07 · Live questioning

The meeting itself — planned one turn at a time

Each run has a budget of 9 questions: 4 intro + 5 dynamic. After every answer the manager types in, the planner runs again and decides what happens next.

Score the answer

What did this answer reveal? The planner nudges the four axes up or down — but only within the limits that question is allowed to move.

Re-plan the queue

Stay on this thread? Drill deeper? Move to the next act of the arc? The planner can add, reword, or drop upcoming questions.

Respect the rules

Caps stop it drilling the same spot forever, asking endless wellbeing clarifiers, or wandering off the arc. When time runs short, it must move forward.

  • Shallow answers don't count as good news. If an answer is just a word or two, any positive scores are zeroed — "fine" is not evidence things are fine.
  • Notes ride along. The manager can jot live notes against any question; they're saved with the run.
  • Everything streams. You can watch the planner think in real time in the web app.
08 · Axes & scoring

Four needles that move as the person talks

Every run tracks four axes. Each answer can nudge them by a small fixed amount (−3, −1, 0, +1, or +3). Scores accumulate over the meeting and are clamped to ±10.

Wellbeing
starts −1
Engagement
starts −1
Clarity
starts 0
Growth
starts 0
How to read a final score: ±0–1 = weak signal, ignore. ±2–4 = worth watching over the coming weeks. ±5–7 = a real pattern, act on it. ±8–10 = a defining signal. Wellbeing and engagement start slightly negative on purpose — they have to be earned by real evidence, not assumed.
09 · The final briefing

What you walk away with

After the last question, Sero reads the whole transcript plus the axis scores and writes the wrap-up:

Headline & summary

One headline (max 22 words), 2–3 bullets, and a short paragraph on how well the person was actually understood.

Axis read-outs

For each axis: what the score means in this conversation and what tone to take about it.

Two brutal truths

One safe to share with the employee. One for the manager's eyes only — never mixed into anything employee-facing.

Three next actions

Each with a "when" attached, so they don't evaporate.

Watch-fors

Two observable signals to keep an eye on between now and the next 1:1.

Honesty switch

If the meeting was thin — short answers, dodged questions — the briefing says so and leads with "we didn't get a good read" instead of pretending to deep insight.

10 · Role profiles

Sero knows the job, not just the person

The first time a role + seniority combo appears (say, "Backend Engineer, mid-level"), Sero generates a role profile once and caches it as a file in data/role-profiles/. Every later run with that combo reuses it for free.

What's in one

A summary of the role, its known challenges, recommended question themes, the terminology people in that job actually use, things to listen for, and things to avoid.

What's NOT in one

Anything personal. Profiles are built from job title + seniority only, so they're safely shared across every person who holds that title.

  • Profiles feed the question bank (so questions sound native to the job) and the preparation briefing (so advice fits the role).
  • The cache key is the exact title — a "Staff SRE" never gets handed a "Backend Engineer" profile.
  • In relational meetings, the profile's evaluative bits are filtered out too — same no-hidden-performance-review rule, and a gate checks it.
11 · The safety gates

Nine hard rules that fail the build if broken

Running npm run gate replays a set of human-approved "golden" test cases through the live pipeline and checks every output against these. Any single failure = the change doesn't ship.

GateWhat it stops
PRIVATE_NOTE_LEAKThe manager's private judgments ("I doubt she's ready") showing up word-for-word in anything the employee could see.
OVERDIAGNOSIS_ON_THINConfident, high scores from a meeting that was actually thin and evasive.
FOCUS_ARC_LEAKA casual check-in or "feels off" meeting sneaking in performance-judging topics.
ROLE_PROFILE_ARC_LEAKThe same sneaking, but via the role profile's evaluative content.
ENGINE_VOCAB_LEAK / ROLE_PROFILE_VOCAB_LEAKInternal machinery words (placeholders, field names like "role_confidence") leaking into what users read.
WRONG_MEETING_TYPEA question bank that ignores the meeting's arc (less than half the acts covered).
SCHEMA_INVALIDOutput that's structurally broken — missing parts, scores out of range.
QUESTION_INTEGRITYDuplicate questions in one meeting, forbidden phrasings, debug text, or questions that appeared from nowhere.
12 · Softer guardrails

Rules baked into the prompts and code

Honesty rules

No silent masking — if the model produces something off, Sero flags it; it never quietly rewrites text to hide problems. Notes with real content must produce at least one note-driven focus point.

Wording rules

Focus-point labels can't start with question words. A banned-phrase list keeps out consultant-speak ("pulse check", "hygiene"). "Best practice" reasons must follow set sentence shapes so they read naturally.

Pacing rules

Max 2 drills in a row on the same spot. Max 2 wellbeing clarifiers in a row. Max 1 off-arc tangent per meeting. When remaining questions ≤ remaining acts, the arc must move forward.

Relevance rules

At least half the question bank must reference something specific to this person — no fully generic banks. Two-word answers can't generate positive scores.

The layering idea: prompts ask the model to behave; code enforces what really matters; and the hard gates catch anything that slips through. Three nets, in that order.
13 · Logs

Every run, fully on file

Each run gets a folder: logs/<month>/<run-id>/ (e.g. logs/june/2026_Jun02_17-35-b4b209e3). Inside, every stage keeps three things: the inputs it got, the exact prompt sent to the model, and the response that came back.

Per stage

inputs.json, prompt.md, response.json — for focus points, preparation, question bank, and evaluation. Plus one file per turn during questioning.

Per run

transcript.json (all Q&A), axis-state.json (final scores), notes.md, session state, and the cost of the run in tokens and dollars.

The fingerprint

pipeline-lock.json records exactly which prompts, code, and models produced this run — so we can tell what changed between any two runs.

One trap to remember: when reviewing a run, response.json is the raw model output, before the guards run. The shipped result is in final.json / the later artifacts — judge the product on what shipped, not the raw draft.
14 · Testing & checking

Three commands tell us if it still works

npm run gate

The trust gate. Replays golden test cases and runs the 9 hard checks. Pass/fail, no opinions. This is the "did we break anything that matters" button — run before calling any change done.

npm run smoke

The end-to-end check. Pushes a whole scenario through the real pipeline, start to finish, to prove the plumbing still connects.

npm run eval

The quality lens. Replays batches and scores soft qualities — how specific the questions are, whether the planner follows threads, whether scores match the transcript.

  • Golden cases live in evals/golden/ — real scenarios a human has ratified, including adversarial ones designed to tempt the model into breaking rules.
  • Scenarios in scenarios/ are replayable personas (a senior backend engineer's weekly, a junior designer's growth chat…) used by smoke and eval.
  • The standing rule: no self-certifying. A prompt or engine change isn't "done" until the gate has actually been run and reported.
15 · The app & the CLI

Two front doors, one engine

The web app

The normal way in. Screens follow the pipeline: Start (with recent runs) → Intake → Focus points → Preparation → Bank → Questioning (live axis bars + notes) → Final briefing → optional Lexicon review. AI stages stream to the page as they generate.

The CLI

npm run cli runs the same pipeline in a terminal — used mostly for testing and replays. It shows a changelog banner when prompts/code/models changed since the last run, and asks for a 1–5 rating at the end.

  • In development: npm run dev starts the API on port 3001 and the web page on port 3000, together. (Under preview tools, run them separately — the port injection clashes.)
  • Sessions survive restarts — a half-finished meeting can be resumed.
  • Light protections: requests only accepted from your own machine; new-session rate limits stop runaway loops.
16 · How we build

The Darren Method

Anything bigger than a one-file tweak gets planned and built one careful step at a time:

1

Plan

A folder under docs/todo/<name>/: a PLAN.md overview plus one file per phase, each ending with test scenarios.

2

Baseline

Run the gate before touching anything, so pre-existing failures don't get blamed on new work.

3

One phase

Build exactly one phase. No racing ahead.

4

You test

The product owner walks the scenarios and gives the green light. The builder never self-certifies.

5

Commit & next

Approved work is committed immediately. Then — and only then — the next phase starts.

Finished work moves to docs/todo/done/ — that folder is the project's build history (role profiles, the meeting-type refactor, demo fixes…). New ideas that pop up mid-phase get parked in PLAN.md, not bolted on.
