03 · The big picture

A product wrapped in a workshop

Two things share one engine. Seeing them apart makes the whole repo click.

The product — a guided 1:1

A manager types rough notes ("she's been quiet"). Sero suggests focus areas, writes a prep briefing, builds a question bank, runs the live conversation scoring as it goes, then writes a final briefing with truths and next actions. This is what a manager touches.

The workshop — keeping it honest

Around it sits build & QA tooling: a run library, compare, per-run verdicts, a regression safety check, the trust gates, an offline eval loop, and a build-plan console. This is how the team trusts the engine before it ships.

One engine, two layouts, every run logged. The manager run can be the classic screen-by-screen flow or the newer one-page grow-down — identical answers and briefing either way. Runs are still logged to the filesystem, but a Postgres foundation now sits underneath, and sign-in fences data by person and org.

05 · The pipeline

Six stages, each with its own model

0

Intake

Name, role, seniority, meeting type, rough notes. No AI.

1

Focus areas

2–5 things to explore, from a catalogue of 21. nano.

1b

Preparation

The game plan: core issue, opener, listen-for, avoid. Full model.

2

Question bank

8–12 tailored questions. mini.

3

Live questioning

Score + re-plan after every answer. Full model, per turn.

4

Final briefing

Headline, truths, next actions, watch-fors. mini.

  • Right model for the job (content/config/models.json): a fast gpt-5.4-nano for focus areas, mini for the bank and evaluation, the full gpt-5.4 for the prep briefing and the live planner.
  • Optional stage 5 — phrase library: Sero can harvest the wording the employee used, to keep future questions in their language.
  • Every stage streams to the screen as it thinks, and logs its inputs, prompt, and reply to disk. Slide 12 lets you read the actual prompts.

07 · Meeting types & arcs

Five kinds of 1:1, each with a shape

Every type has an arc — a planned run of stages, like acts in a play, each with a target number of questions. The planner moves the conversation through it.

Bi-weekly check-inCasual catch-up, 15–20 minpulse → friction → momentum → liftRelational: no performance-judging topics
Performance & feedbackDirect, adult-to-adultself-read → evidence → gap → cause → commit—
Growth & career planForward-looking, 30–45 minanchor → aspiration → gap → investment → commitmentIts own final-briefing voice

Arcs are browsable & editable from the "Meeting arcs" page. The relational rule — no competency topics in a casual check-in — is enforced at four layers: the input filter, the prompt, the focus gate (FOCUS_ARC_LEAK), and the question-layer gate (QUESTION_ARC_LEAK).
08 · Focus areas

What the meeting is really about — 21 on the menu

Sero picks from a fixed catalogue of 21 focus areas (content/focus-points.json), each with worked examples so the wording fits the level:

Wellbeing — 3

Workload & capacity, energy & wellbeing, team connection. How the person is actually doing.

Topic — 7

Priorities, blockers, growth, feedback, recognition, role clarity, manager support.

Competency — 11

Quality, speed, ownership, communication, reliability, judgment, impact, decision speed, technical problem-solving, stakeholder engagement, delegation. Off-limits in relational meetings.

  • Each is a signal (drawn from your notes) or a best practice (sensible for this role anyway) — and carries a reason and a confidence.
  • If your notes have real content, at least one suggestion must come from them — Sero can't ignore what you wrote.
  • You start with nothing selected; your picks steer the whole question bank.
09 · The preparation briefing

The manager's game plan

Once focus areas are chosen, Sero writes a short strategy — a fixed shape you can lean on:

Core issue

What this 1:1 is really about, in one sentence — by name, never job title.

Opening question

A line the manager can say verbatim to open it up.

Listen for

Three behavioural tells — deflections, pivots, energy shifts.

Avoid

Two traps likely to shut this conversation down.

Good outcome

The one concrete agreement reachable in this meeting.

Confidence + don't-assume

How much the read rests on, and the tempting verdict the data does not yet support.

Quality control built in: the briefing is validated before you see it; if it fails, Sero retries once. It refers to people by name, in plain language — no jargon, no clinical labels. (You can read this exact prompt on slide 12.)
10 · The question library

4,375 questions, each its own small file

Every question Sero has generated and kept lives as a tiny YAML file in content/questions/. Each records what it says and how it behaves:

What it says

The text, a short label, a plain description, and its purpose: wellbeing, topic, or competency.

How it behaves

Its arc stage, which axes it can move and by how much (its "signature"), and where it came from.

  • The index (content/questions/_index.json) is the master list — rebuilt with npm run rebuild-question-index.
  • Openers & seeds are kept separately: research-backed openers plus per-type intro questions to start each run.
  • The dedup gate stops near-identical questions in a session — it compares the actual words, not the filename.
  • Questions stay in their own run: ones invented mid-meeting save to a separate runtime area, never leaking across sessions.
11 · Live questioning

The runner — planned one turn at a time

A run has a budget of 9 questions: 4 intro + 5 dynamic. After each answer the manager types, the planner recomputes its runtime signals and decides what's next.

What it tracks each turn

How far through the arc, which stages remain, drills-in-a-row at one spot, wellbeing clarifiers in a row, off-arc tangents used, whether it's the final turn, and the last answer's scores.

What it decides

Stay on this thread, drill deeper, or advance the arc — adding, rewording, or dropping upcoming questions to fit.

Hard caps in code

≤2 drills in a row, ≤2 wellbeing clarifiers in a row, ≤1 off-arc tangent. When questions left ≤ acts left, it must advance. Crisis disclosure clears the queue for one warm support question.

  • Shallow answers don't count as good news. A one- or two-word answer (or a skip) gets any positive scores zeroed — "fine" is not evidence things are fine.
  • Grounding gate: a brand-new question must quote a real premise from the notes or transcript — invented premises get dropped.
  • Want to see exactly what it's sent each turn? → next slide.
12 · The prompt viewer INTERACTIVE

This isn't a mock-up. The text in each panel is the actual prompt from content/prompts/ (long rule blocks trimmed with a marker). In the app itself, the "Sent" tab shows the fully-filled version for a real run, byte-for-byte, before it's sent.
13 · Axes & scoring

Four needles, all starting at zero

Wellbeing · Engagement · Clarity · Growth. Every run starts all four at 0 — neutral. Each answer nudges them by a fixed step (−3, −1, 0, +1, or +3), bounded by what that question is allowed to move; scores accumulate and clamp to ±10.

−10…−5 act
−4…−2 watch
±1 weak
+2…+4 watch
+5…+10 strong
−100 · start here+10
±0–1 — weak signal, not actionable. Ignore.
±2–4 — worth watching over the coming weeks.
±5–7 — a real pattern. Act on it.
±8–10 — a defining signal of this conversation.
Neutral start = nothing assumed. A positive or negative read has to be earned by what's said. And a score built from just one or two mentions gets its confidence dropped a tier — magnitude alone doesn't earn the loudest verdict.
14 · The final briefing

What you walk away with

After the last question, Sero reads the whole transcript plus the axis scores and writes the wrap-up:

Headline & summary

One headline (≤22 words), 2 synthesis bullets, and a short paragraph on what was newly understood.

Axis read-outs

For each axis: what the score means here and what tone to take about it.

Two brutal truths

One safe to share; one manager's-eyes-only — never mixed into anything employee-facing.

Two next actions + reminders

Each timed and concrete; plus two paste-ready "watch-for" reminders.

Engagement read

An honest "how engaged are they" that stays inconclusive unless the evidence truly earns a concern.

Honesty switches

If the meeting was thin, the headline leads with that. Never fakes insight from skipped turns.

"Thin" has a precise meaning: fewer than 3 substantive turns, or under 60% of turns carrying real signal. Then the briefing shrinks and one action becomes "have the conversation again." (Read this prompt on slide 12 — it's the most rule-dense of the five.)
15 · Role profiles & vocabulary

Sero knows the job, not just the person

The first time a role + seniority appears, Sero generates a role profile once and caches it in content/data/role-profiles/. Later runs reuse it for free. 24 are on file now.

What's in one

A summary, known challenges, recommended question themes, the terminology people in that job use, things to listen for, and things to avoid.

What's NOT in one

Anything personal. Built from title + seniority only — safely shared across everyone with that title. The cache key is the exact title.

  • Profiles feed the question bank and the prep briefing so both sound native to the job.
  • The terminology is surfaced to you — as the "language of this role" glossary and on the Job Lexicons page.
  • In relational meetings the evaluative parts are filtered out, with a gate (ROLE_PROFILE_ARC_LEAK) to catch leaks.
16 · Job lexicons

Browse — and add to — each job's words

A left-rail page lists every role's vocabulary: the AI's terms with plain meanings. You can also teach Sero your own job words.

Browse anytime

No active meeting needed. See each title with its words — what "snagging" or "handoff" means in that world.

Add your own

Add a term to any role. Yours are marked "yours" and saved to a separate overlay file — never overwriting the AI's, removable in one click.

  • Your words show up in the "language of this role" glossary during a run.
  • Distinct from the phrase library (which harvests wording from one past session). Job lexicons are permanent, per-role, editable.
  • The point: Sero should sound like it works on your team, not like a generic HR tool.
17 · The safety gates

Eleven hard rules that fail the build

npm run gate replays human-approved "golden" cases through the live pipeline. Any one of these failing means the change doesn't ship (evals/trust-checks.js).

GateWhat it stops
PRIVATE_NOTE_LEAKThe manager's private judgments surfacing word-for-word where the employee could see them.
OVERDIAGNOSIS_ON_THINConfident, high scores from a thin, evasive meeting.
FOCUS_ARC_LEAK / QUESTION_ARC_LEAKA relational meeting sneaking in performance-judging topics — caught at the focus and question layers.
ROLE_PROFILE_ARC_LEAKThe same sneaking, but via the role profile's evaluative content.
CROSS_SESSION_QUESTION_LEAKA run serving a question or vocabulary from someone else's session.
ENGINE_VOCAB_LEAK / ROLE_PROFILE_VOCAB_LEAKInternal machinery words leaking into what users read.
WRONG_MEETING_TYPEA question bank that ignores the meeting's arc.
SCHEMA_INVALIDStructurally broken output — missing parts, scores out of range.
QUESTION_INTEGRITYDuplicate questions, forbidden phrasings, debug text, or questions from nowhere.
18 · Engine trust gates

Six protections, one per problem found

A review of past runs turned up five recurring engine bugs. Each was fixed and locked behind a check:

#ProtectionWhat it now guarantees
1Question isolation
CROSS_SESSION_QUESTION_LEAK		Mid-meeting questions save to a separate runtime area — never reappearing in an unrelated run.
2Honest follow-up stemsA "you said…" follow-up must quote a real, contiguous phrase; no clean phrase → it's skipped, not faked.
3Grounded premises
UNGROUNDED_PREMISE		Every new question cites a short verbatim quote from notes/transcript, or is marked open. Invented premises drop.
4Relational-arc at the question layer
QUESTION_ARC_LEAK		Bi-weekly and "feels off" can't generate, pick, or add a competency question.
5Axis honesty
AXIS_SILENT_SESSION		Carried-forward questions keep their scoring power, so a real conversation doesn't ship every axis as "didn't come up".
6Confidence honesty
RULE_ECHO_MEANING		A high score on ≤2 mentions gets its confidence downgraded; meanings that echo a prompt example are flagged, never silently rewritten.
The through-line: say what's actually true. Don't borrow questions, invent premises, fake confidence, or paper over a problem.
19 · Guardrails & advisory

Three nets, in order

Prompts ask the model to behave; code enforces what really matters; the gates catch anything that slips. Some checks are advisory — they warn and log rather than fail the build.

Prompt-level rules

Focus labels can't start with question words. A banned-phrase list keeps out consultant-speak. Notes with content must yield ≥1 note-driven focus. ≥half the bank must reference this specific person.

Code-level enforcement

Drill / clarifier / tangent caps. Shallow-answer zeroing. The grounding gate. The arc-leak filter. Safe fallbacks for broken questions. These don't ask — they enforce.

Advisory (warn + log)

UNGROUNDED_PREMISE, AXIS_SILENT_SESSION, RULE_ECHO_MEANING — surfaced for us to inspect, not hard-blocking yet. 14 named checks: 11 hard-fail + 3 advisory.

The standing rule

No silent masking. When the model produces something off, Sero flags it — it never quietly rewrites text to hide a problem. Surfacing the truth beats a clean-looking lie.

Per stage

Per run

21 · The QA cockpit

How the team keeps the engine honest

A big slice of the console is tooling for reviewing runs and turning what we learn into prompt fixes. The loop:

Library — the inbox

Every finished run, newest first. Filter by verdict, search, archive. A bar tracks how many are reviewed.

Review — the verdict

Read a run end-to-end and mark 8 dimensions Pass/Fail, then Keep / Fix / Block. Saved to review.json.

Compare — side by side

Load two runs together to see what a prompt change moved: headlines, questions, notes, axis reads with delta bars.

Suggest-fix

With a verdict recorded, ask the model for a minimal edit to one stage's prompt. Display-only.

Regression — instant safety

Re-checks saved "key runs" against the current engine with no AI calls. Catches a prompt change breaking a safety rule in seconds.

Notes & debrief

A notes panel tags observations to the stage/turn they happened on; the debrief shows time, cost, and turn stats.

Who sees what: most of these screens are for the team. The manager only ever sees the run itself.

23 · Testing & the eval loop

What proves it still works

npm run gate

The trust gate. Replays 8 golden scenarios and runs all hard checks. Pass/fail. Before any change is "done". ~$3 full, ~$0.35 a single case.

npm run smoke

End-to-end check. Pushes a whole scenario start to finish to prove the plumbing connects.

npm test / typecheck

Free & fast. ~69/69 unit tests plus a strict TypeScript check — the everyday safety net.

  • The prompt-notes loop: human-written rules in *.notes.yaml (must/must-not, length & count ranges) are checked against output offline. Fail → read why, tweak the prompt, re-check — without burning API.
  • Free offline nets: npm run replay (7 regression fixtures) and offline trust-check tests. These are the default.
  • Cost discipline: no paid run without an explicit yes and a stated cost first. No self-certifying — the gate runs and is reported before anything is called done.
24 · How we build

The Darren Method

Anything bigger than a one-file tweak is planned and built one careful step at a time:

1

Plan

A folder under docs/todo/<name>/: a PLAN.md plus one file per phase, each ending in test scenarios.

2

Baseline

Run checks before touching anything, so old failures aren't blamed on new work.

3

One phase

Build exactly one phase. No racing ahead.

4

You test

The product owner walks the scenarios and green-lights. The builder never self-certifies.

5

Commit & next

Approved work commits immediately. Only then does the next phase start.

One board tracks it all: SERO_BOARD.md is the single source of truth — Now / Next / Parked / Cut / Done, with commit hashes. Finished folders move to docs/todo/done/; mid-phase ideas get parked, never bolted on.
